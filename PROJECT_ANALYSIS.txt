PURPOSE
-------
A full-stack, real-time private messaging web application with AI-powered
assistance. Users can register, log in, select any other registered user, and
exchange private messages that are delivered instantly without page refresh.
The app also uses Google Gemini to provide AI smart-reply suggestions and
AI-powered chat summarization.

PROBLEM IT SOLVES
-----------------
Traditional messaging relies on polling (repeated HTTP requests) which is slow,
wasteful, and creates poor UX. This app uses WebSockets (Socket.IO) to push
messages to clients the instant they are sent — just like WhatsApp or Slack.
The AI layer reduces the cognitive load of typing replies by suggesting short
contextual responses.

TARGET USERS
------------
- Students learning full-stack development (this is a portfolio project)
- Anyone demonstrating real-time web app skills
- Teams wanting a self-hosted private messaging solution

KEY FEATURES
------------
1. User registration and login with bcrypt-hashed passwords
2. JWT-based stateless authentication for both REST API and WebSocket
3. Real-time private messaging via Socket.IO rooms
4. Persistent message history stored in MongoDB (loads on chat open)
5. AI Smart Replies — click the ✨ button to get 3 AI-generated reply options
6. AI Chat Summary — click the 📖 button to summarize the conversation
7. Glassmorphism UI with dark-mode support and framer-motion animations
8. Responsive layout: 3-panel (desktop), 2-panel (tablet), 1-panel (mobile)
9. Contact search with real-time filtering

SECTION 2 — HOW THE SYSTEM WORKS (STEP BY STEP FLOW)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SIGNUP FLOW
-----------
1. User fills out username, email, password in Signup.jsx
2. Frontend makes POST /api/auth/signup via Axios
3. authController.js checks if email already exists in MongoDB
4. Password is hashed using bcryptjs (salt rounds = 10)
5. New User document is saved to MongoDB
6. Response: 201 { message: "Signup successful" }
7. User is redirected to Login page

LOGIN FLOW
----------
1. User enters email + password in Login.jsx
2. Frontend makes POST /api/auth/login via Axios
3. authController.js finds user by email in MongoDB
4. bcrypt.compare() checks plain password against the stored hash
5. If match: JWT token is signed with user._id, expires in 7 days
6. Response: 200 { token, user: { id, username, email } }
7. Frontend stores token + user object in localStorage
8. User is navigated to /chat

CHAT PAGE LOAD FLOW
-------------------
1. Chat.jsx mounts, reads token from localStorage
2. If no token → redirect to /login
3. Socket.IO connection established: io(BACKEND_URL, { auth: { token } })
4. Backend Socket.IO middleware verifies the JWT
5. GET /api/users is called — returns all users EXCEPT the logged-in user
6. Sidebar displays the user list

SELECT A USER TO CHAT
---------------------
1. User clicks a contact in Sidebar
2. joinChat(user) called in Chat.jsx
3. A unique roomId is computed: [myId, otherId].sort().join("_")
   (Sorting ensures same roomId regardless of who initiates)
4. socket.emit("join-room", roomId) → backend calls socket.join(roomId)
5. GET /api/messages/:roomId → loads full chat history from MongoDB
6. Messages displayed in ChatWindow.jsx, auto-scrolled to bottom

SENDING A MESSAGE
-----------------
1. User types in input, presses Enter or clicks Send
2. sendMessage() in Chat.jsx fires socket.emit("send-message", { roomId, message, receiver })
3. Backend socket handler:
   a. Saves Message to MongoDB (sender, receiver, content, roomId, timestamp)
   b. Emits "receive-message" to all OTHER sockets in that room
4. The sender side: message is optimistically added to local state
   (sender: "me") immediately — no waiting for server echo
5. The receiver side: socket.on("receive-message") appends the message

AI SMART REPLY FLOW
-------------------
1. User clicks the ✨ (Sparkles) button in the input area
2. handleRequestSmartReplies() in Chat.jsx fires
3. Finds the last received message (sender !== "me")
4. POST /api/ai/smart-reply { lastMessage } with JWT header
5. aiRoutes.js passes the message to Gemini with prompt:
   "Give exactly 3 short friendly chat replies for: '...'"
6. Gemini returns text, backend parses into 3 reply strings
7. Frontend shows animated chip buttons above the input
8. Clicking a chip fills the message input for review before sending

AI SUMMARIZE FLOW
-----------------
1. User clicks the 📖 (BookOpen) button in the chat header
2. handleSummarize() in Chat.jsx fires
3. Formats all messages as [{ sender: "username", message: "..." }]
4. POST /api/ai/summarize { messages } with JWT header
5. Backend builds a chat log string, sends to Gemini with prompt:
   "Summarize this chat in 2-3 concise sentences"
6. Summary displayed in an animated purple banner under the header

LOGOUT FLOW
-----------
1. User clicks logout in Sidebar
2. localStorage cleared (token + user)
3. socket.disconnect() terminates the WebSocket
4. User navigated to / (Login page)

API ROUTES — FULL REFERENCE
----------------------------

POST /api/auth/signup
  Body    : { username, email, password }
  Response: 201 { message: "Signup successful" }
  Auth    : None
  Logic   : Validates fields → check duplicate email → bcrypt hash → save User

POST /api/auth/login
  Body    : { email, password }
  Response: 200 { token, user: { id, username, email } }
  Auth    : None
  Logic   : findOne(email) → bcrypt.compare → jwt.sign({ id }, SECRET, 7d)

GET /api/users
  Response: 200 [ { _id, username, email }, ... ] — excludes self
  Auth    : Bearer JWT (required)
  Logic   : User.find({ _id: { $ne: req.userId } }, "username email")

GET /api/messages/:roomId
  Response: 200 [ { _id, sender: {_id, username}, receiver, content, roomId, createdAt }, ... ]
  Auth    : Bearer JWT (required)
  Logic   : Message.find({ roomId }).populate("sender", "username").sort(createdAt: 1)

POST /api/messages (not used by frontend currently — messages saved via Socket.IO)
  Body    : { receiver, content, roomId }
  Response: 201 { message document }
  Auth    : Bearer JWT (required)

POST /api/ai/smart-reply
  Body    : { lastMessage: "string" }
  Response: 200 { replies: ["reply1", "reply2", "reply3"] }
  Auth    : Bearer JWT (required)
  Logic   : Gemini prompt → parse 3 lines → return array

POST /api/ai/summarize
  Body    : { messages: [{ sender: "name", message: "text" }] }
  Response: 200 { summary: "2-3 sentence summary" }
  Auth    : Bearer JWT (required)
  Logic   : Build chat log string → Gemini prompt → return trim()

SOCKET.IO EVENTS — FULL REFERENCE
-----------------------------------
Connection auth:  token in socket.handshake.auth.token — verified via JWT middleware

Client → Server Events:
  "join-room"    payload: roomId (string)
                 Action: socket.join(roomId) — socket now receives messages in that room

  "send-message" payload: { roomId, message, receiver }
                 Action: Saves Message to MongoDB → emits "receive-message" to room

Server → Client Events:
  "receive-message"  payload: { message, sender (userId), roomId }
                     Triggered: When another socket in the room sends a message

  "error"            payload: { message: "..." }
                     Triggered: If message save fails

Server-side state:
  onlineUsers : Map<userId, socketId>  — tracks who is currently connected

AUTHENTICATION FLOW — DETAILED
--------------------------------
REST API:
  Header: "Authorization: Bearer <JWT>"
  Middleware (in each encrypted route file):
    1. token = req.headers.authorization?.split(" ")[1]
    2. decoded = jwt.verify(token, process.env.JWT_SECRET)
    3. req.userId = decoded.id → available in route handler
    4. On failure: 401 (no token) or 403 (invalid/expired)

Socket.IO:
  1. Client passes token in: io(URL, { auth: { token } })
  2. Server middleware: io.use((socket, next) => { ... })
  3. token = socket.handshake.auth?.token
  4. jwt.verify() → socket.userId = decoded.id
  5. On failure: next(new Error("Invalid token")) → socket closed

JWT Payload: { id: user._id }  — keeps token lean
JWT Secret:  process.env.JWT_SECRET
JWT Expiry:  7 days

STATE MANAGEMENT APPROACH
--------------------------
No external state library. Pure React useState + useMemo.

Key pattern: messagesByRoom
  { "userId1_userId2": [ {message, sender, _id}, ... ] }
  → Allows switching between chats without losing history already loaded
  → useMemo stabilizes the derived `messages` array reference to prevent
    spurious re-renders (the || [] fallback issue was the source of the
    infinite render loop that was later fixed)

ERROR HANDLING
--------------
Backend:
  - try/catch in all async route handlers
  - Returns { message: err.message } with appropriate status codes
  - Socket sends "error" event on message save failure
  - AI routes log "GEMINI ERROR" and return 500 with { error: "..." }

Frontend:
  - Axios catch blocks for all API calls
  - Login failure → alert + redirect to signup
  - AI quota/rate limit (429→500) → shows "⚠️ AI quota exceeded" message
  - Loading states on all async operations (loading, smartRepliesLoading, summaryLoading)
  - Socket error logged to console

WHY THESE TECHNOLOGIES?
------------------------
React + Vite
  → Vite offers near-instant HMR (Hot Module Replacement) vs Create React App
  → Component model fits perfectly for chat UI (Sidebar, ChatWindow, etc.)
  → Alternative: Next.js (adds SSR), Angular (more opinionated), Vue

Socket.IO
  → Abstracts WebSocket with automatic fallback (polling if WS blocked)
  → Built-in room management (socket.join/to) — perfect for private chats
  → Auto-reconnect handling built in
  → Alternative: raw WebSocket API, Pusher (managed), Ably

MongoDB + Mongoose
  → Flexible schema = easy to iterate; messages are just documents
  → MongoDB Atlas = free hosted DB for projects
  → Mongoose adds schema validation, populate() for joins, timestamps
  → Alternative: PostgreSQL + Prisma (more relational), Redis for sessions

JWT Authentication
  → Stateless — server doesn't need to store sessions
  → Works for both HTTP (Authorization header) and WebSocket (handshake)
  → Alternative: sessions + cookies (stateful), Auth0 (managed)

bcryptjs
  → Industry-standard password hashing with salt (prevents rainbow table attacks)
  → Salt rounds = 10 (balance between security and performance)
  → Alternative: argon2 (newer, slightly more secure), scrypt

Tailwind CSS v4
  → Utility-first = no context switching between CSS files and JSX
  → v4 uses @import instead of PostCSS config
  → CSS variables + @theme for dynamic light/dark theming
  → Alternative: CSS Modules, styled-components, vanilla CSS

Framer Motion
  → Declarative animations perfect for React
  → AnimatePresence handles mount/unmount transitions (AI banner, chips)
  → Alternative: React Spring, CSS transitions, GSAP

Google Gemini API
  → Free tier available, very capable for chat-assistance tasks
  → gemini-2.0-flash is fast and lightweight
  → Alternative: OpenAI GPT-4o-mini, Anthropic Claude

DESIGN DECISIONS
-----------------
1. ROOM ID STRATEGY
   roomId = [user1._id, user2._id].sort().join("_")
   WHY: Alphabetical sort ensures that the room between A and B is always
   "A_B" regardless of who opens the chat first. No duplicates.

2. OPTIMISTIC UI
   When sending, the message is added to local state IMMEDIATELY with
   sender: "me", before waiting for any server confirmation.
   WHY: Feels instant. If socket fails, it should still be recovered via
   socket error handling.

3. MESSAGES STORED IN SOCKET HANDLER (not REST POST)
   Messages are saved inside the "send-message" socket handler instead of
   calling the REST API.
   WHY: Eliminates an extra HTTP round-trip. The WebSocket connection is
   already open — use it for everything.

4. MESSAGES KEYED BY ROOM (not flat array)
   { "room1": [], "room2": [] } vs [ all messages ]
   WHY: When user switches chats, already-fetched messages are cached.
   No redundant API calls.

5. AI ON-DEMAND (not automatic)
   Smart replies trigger only on button click, not on every message.
   WHY: Prevents burning through the Gemini free tier quota (429 errors).
   Respects the user's intention to ask for help.

SECURITY CONSIDERATIONS
------------------------
1. Passwords never stored in plain text — bcrypt hashed with salt
2. JWT signed with server secret — never stored in JWT itself
3. Auth middleware on EVERY protected route — no accidental open endpoints
4. Socket.IO auth middleware — even WebSocket connections require valid JWT
5. CORS restricted to frontend origin only (localhost:5173)
6. MongoDB $ne query excludes self from user list server-side
7. No SQL injection possible (NoSQL + Mongoose validation)

SECURITY GAPS (for production, would need to fix):
- Token stored in localStorage (vulnerable to XSS) → use httpOnly cookies
- No token refresh mechanism (token just expires after 7 days)
- No rate limiting on auth endpoints (brute force risk) → add express-rate-limit
- No input sanitization (XSS in message content possible)
- No HTTPS enforcement
- userRoutes.js missing try/catch around jwt.verify (will crash on bad token)
- No message pagination (loading all history at once won't scale)

BASIC QUESTIONS (Easy)
════════════════════════════════════

Q1: What does this project do?
A: It's a real-time private messaging web app with AI assistance. Users can
   sign up, log in, and chat with any other registered user. Messages are
   delivered instantly using WebSockets. The app also has AI features powered
   by Google Gemini — smart reply suggestions and chat summarization.

Q2: What is the tech stack?
A: Frontend: React 19 with Vite, Tailwind CSS v4, Framer Motion, Socket.IO client,
   Axios. Backend: Node.js with Express.js, Socket.IO server, MongoDB with
   Mongoose, JWT + bcryptjs for auth, Google Gemini API for AI.

Q3: What is a WebSocket and why did you use it?
A: A WebSocket is a persistent, bidirectional connection between client and
   server. Unlike HTTP which is request-response (you ask, server responds,
   connection closes), WebSocket stays open. This means the server can PUSH
   data to the client anytime — without the client asking. In chat, this means
   messages appear instantly on the receiver's screen without polling.

Q4: What is the difference between REST and Socket.IO in this app?
A: REST is used for operations that don't need real-time push: login, signup,
   fetching user list, fetching message history. Socket.IO is used for the
   real-time message delivery — when a user sends a message, Socket.IO pushes
   it to the receiver immediately. Mixing both is the standard architecture.

Q5: How do you store passwords?
A: Never in plain text. bcryptjs hashes the password with a salt of 10 rounds.
   Even if the database is compromised, attackers can't reverse-engineer the
   passwords. bcrypt.compare() is used at login to verify without decrypting.

Q6: What is JWT and how do you use it?
A: JWT (JSON Web Token) is a signed token that contains a payload — in this
   case, just the user's ID. After login, the server creates a token with
   jwt.sign({ id: user._id }, SECRET, { expiresIn: "7d" }) and sends it to
   the client. The client stores it in localStorage and includes it in every
   request header as "Authorization: Bearer <token>". The server verifies it
   with jwt.verify() without needing to query the database.

INTERMEDIATE QUESTIONS
════════════════════════════════════

Q7: How does a private chat room work? How do you ensure User A and User B
    have the same room?
A: The roomId is computed as [userId_A, userId_B].sort().join("_"). Sorting
   alphabetically ensures the ID is always identical regardless of who opens
   the chat first. For example, if A = "abc" and B = "xyz", the room is
   always "abc_xyz" — never "xyz_abc". Both users emit "join-room" with this
   ID, and Socket.IO's socket.join(roomId) adds them to that room. Messages
   emitted to that room only go to sockets in it.

Q8: What is the messagesByRoom pattern and why use it?
A: Instead of a flat messages array, I use an object keyed by roomId:
   { "roomId_1": [...], "roomId_2": [...] }. This caches messages per chat.
   When a user switches between conversations, already-loaded messages are
   instantly available without making another API call. The current messages
   are derived with useMemo(() => messagesByRoom[currentRoom] || [], [...]).

Q9: You mentioned an infinite re-render bug. Can you explain it?
A: Yes. Originally, messages was derived as:
   const messages = messagesByRoom[currentRoom] || [];
   The || [] creates a NEW empty array on every render (arrays are reference
   types in JS). So React saw "messages changed" on every render, triggering
   the useEffect that called fetchSmartReplies, which set state, which caused
   another render, and so on — an infinite loop. The fix was wrapping it in
   useMemo() so the array reference is stable as long as messagesByRoom and
   currentRoom haven't actually changed.

Q10: How does the Socket.IO authentication work?
A: Before any socket can connect, the backend runs an auth middleware:
   io.use((socket, next) => { ... }). It reads the token from
   socket.handshake.auth.token, verifies it with jwt.verify(), attaches the
   decoded userId to socket.userId, and calls next(). If the token is missing
   or invalid, it calls next(new Error(...)) which rejects the connection.
   This means you can't establish a WebSocket without a valid JWT — the same
   security model as the REST API.

Q11: How do messages persist across page refreshes?
A: When a user selects a chat partner, loadChatHistory() calls
   GET /api/messages/:roomId. The backend queries MongoDB for all messages
   with that roomId, populates the sender's username, sorts by createdAt
   ascending, and returns them. These are formatted and stored in
   messagesByRoom. Socket.IO only handles NEW messages in the current session.

Q12: Explain the component hierarchy of the frontend.
A: main.jsx renders App inside BrowserRouter. App defines routes. /chat
   renders Chat.jsx which owns ALL state and logic. Chat.jsx passes everything
   down to ChatLayout.jsx via props. ChatLayout manages the responsive
   3/2/1 panel structure and passes focused props to Sidebar, ChatWindow,
   and UserInfo. This is a top-down data flow (one-way binding). ChatWindow
   contains the messages and input, Sidebar has the contact list.

ADVANCED / SYSTEM DESIGN QUESTIONS
════════════════════════════════════

Q13: How would you scale this app to support 1 million concurrent users?
A: Several layers:
   1. HORIZONTAL SCALING: Run multiple Node.js instances behind a load
      balancer (Nginx/AWS ELB). But Socket.IO rooms don't work across
      instances by default.
   2. SOCKET.IO ADAPTER: Use socket.io-redis-adapter so all instances share
      a Redis pub/sub channel. A message emitted on instance 1 propagates
      to instance 2's sockets.
   3. DATABASE: Add MongoDB indexes on roomId + createdAt. Use connection
      pooling. Add read replicas for history queries.
   4. MESSAGE PAGINATION: Load last 50 messages, add infinite scroll upward.
   5. CDN: Serve static frontend assets via CDN (Cloudflare, Vercel).
   6. CACHING: Cache user list in Redis (invalidate on new signup).
   7. MESSAGE QUEUE: Use RabbitMQ or Kafka for guaranteed message delivery.

Q14: Why store messages in the socket handler instead of calling the REST API?
A: It eliminates an extra HTTP round-trip. The socket connection is already
   open and authenticated. Adding a REST call would:
   a) Take extra time (new HTTP connection or keep-alive overhead)
   b) Require auth header duplication
   c) Add complexity with no benefit since the socket auth already validates
   The trade-off: if the message DB save fails, the sender sees an error but
   already added the message optimistically. This requires a retry mechanism
   in production.

Q15: What is the problem with storing JWT in localStorage?
A: localStorage is accessible via JavaScript, making it vulnerable to XSS
   (Cross-Site Scripting) attacks. If an attacker injects JS into the page,
   they can read the token. The safer approach is to store JWT in an httpOnly
   cookie, which JavaScript cannot read — only the browser sends it
   automatically with each request. For production: use httpOnly + Secure +
   SameSite=Strict cookies, and implement CSRF protection.

Q16: How does the AI integration work end-to-end?
A: The frontend calls POST /api/ai/smart-reply or /api/ai/summarize with a
   JWT. The backend uses the @google/generative-ai SDK to initialize Gemini
   (lazy singleton via getGenAI()). It sends a crafted prompt to the Gemini
   model. For smart replies: parses the response text into 3 strings by
   splitting on newlines and stripping numbering. For summarize: returns the
   trimmed response text. The frontend displays results as animated chips or
   a banner. Rate limiting (429) is handled gracefully with a user-friendly
   error message.

DEBUGGING QUESTIONS
════════════════════════════════════

Q17: A user says messages aren't appearing in real time. How do you debug?
A: Systematic approach:
   1. Check browser Network tab → WebSocket frames. Is the connection open?
   2. Check if the "join-room" event was emitted with the correct roomId.
   3. Add console.log on the server's "send-message" handler — is it firing?
   4. Check if the receiver's socket is in the same room (socket.rooms Set).
   5. Check if "receive-message" is reaching the client side.
   6. Verify both users have the same roomId string (sort order consistent?).
   7. Check for CORS errors in the socket connection.
   8. Verify the JWT is valid for the receiver's socket connection.

Q18: The AI endpoint returns 500. How do you debug?
A: First, check the server console for "GEMINI ERROR:" logs — the error
   message is printed there. Common causes seen in this project:
   - Model name not available for the API key tier (e.g., gemini-2.5-flash
     not available on free tier → switched to gemini-2.0-flash)
   - 429 rate limit exceeded (free tier quota exhausted)
   - Missing or invalid GEMINI_API_KEY in .env
   - Network issue from server to Google's API
   Fix: Log err.message AND err.status to distinguish these cases.

Q19: How do you handle the case where a user's token expires?
A: Currently, if a token expires, the API returns 403 Forbidden. The
   frontend axios calls fail, and the user sees errors. In production, you'd:
   1. Use an axios response interceptor to catch 401/403 globally
   2. Clear localStorage and redirect to /login
   3. Or implement token refresh: issue a short-lived access token +
      long-lived refresh token stored in httpOnly cookie.

Q20: What are the biggest bottlenecks in the current architecture?
A: 1. Single server process — can't scale horizontally without Redis adapter
   2. No message pagination — loading 10,000 messages at once is catastrophic
   3. No indexes on messages collection — full collection scan on every load
   4. onlineUsers Map in memory — lost on server restart
   5. No rate limiting — auth endpoints vulnerable to brute force
   6. Synchronous AI calls block the response — could use queuing
   7. localStorage JWT — security concern at scale

   - Add a "Conversation" model: { members: [ObjectId], type: "private"|"group", name: String }
   - Add roomId to reference a Conversation document
   - Modify the roomId strategy: for groups, use conversationId directly
   - Add group creation API endpoint
   - Modify the Sidebar to show both direct messages and groups
   - Socket.IO already supports rooms of any size — no changes needed there

DATABASE DESIGN QUESTIONS
════════════════════════════════════

Q24: Why MongoDB over a relational database for this project?
A: For this project, MongoDB's flexible schema works well. Messages are
   documents with predictable, simple structure. There's minimal relational
   complexity. MongoDB Atlas provides a free hosted tier. However, for a
   production chat app at scale, a relational DB (PostgreSQL) might be better
   because: strict schema validation, better JOIN performance for complex
   queries (e.g., "last message per conversation"), ACID transactions for
   message delivery guarantees.

Q25: What indexes would you add to the messages collection?
A: { roomId: 1, createdAt: 1 } — compound index. Supports:
   - Message.find({ roomId }).sort({ createdAt: 1 }) → exact match + sort
   - Pagination: .find({ roomId, createdAt: { $gt: lastSeen } })
   Additional: { sender: 1 } and { receiver: 1 } for user-specific queries.

Q26: How would you implement message read receipts?
A: Add a "readAt" field to Message model: readAt: { type: Date, default: null }
   When receiver opens the chat / sees the message, emit a "message-read"
   socket event with the message IDs. Server updates Message.readAt = new Date()
   Sender's frontend subscribes to "message-read" and updates the UI with "✓✓".

AUTHENTICATION & SECURITY QUESTIONS
════════════════════════════════════

Q27: What is bcrypt salt and why is it important?
A: A salt is a random string prepended to the password before hashing.
   Without salt, two users with the same password get the same hash — enabling
   "rainbow table" attacks (pre-computed hash → password lookups). With a salt,
   even identical passwords produce completely different hashes. bcrypt
   automatically generates and stores the salt inside the hash string, so you
   don't need to store it separately. Salt rounds = 10 means 2^10 = 1024
   iterations, making brute force very slow.

Q28: How would you implement role-based access control (RBAC)?
A: Add a "role" field to the User schema: role: { type: String, enum: ["user", "admin"], default: "user" }
   Include role in the JWT payload. Create a middleware factory:
   const requireRole = (role) => (req, res, next) => {
     if (req.userRole !== role) return res.sendStatus(403);
     next();
   }
   Apply to admin routes: router.delete("/messages", auth, requireRole("admin"), ...)

"Let me walk you through my real-time chat application.

The core challenge in any chat app is real-time message delivery. I solved
this with Socket.IO, which provides WebSocket connections with automatic
fallback. When a user logs in, the frontend immediately establishes a
persistent socket connection — passing the JWT token during the handshake.
On the server side, I have a Socket.IO auth middleware that verifies this
JWT, so even the WebSocket layer is secured.

For private chats, I use a room-based strategy. When User A opens a chat
with User B, both client and server compute a roomId by sorting their
MongoDB ObjectIDs alphabetically and joining them. This guarantees the room
ID is always the same regardless of who initiates. Both clients join this
room, and messages are emitted only to sockets within that room using
socket.to(roomId).emit().

For persistence, when a user opens a conversation for the first time in a
session, the frontend calls a REST endpoint — GET /api/messages/:roomId —
which queries MongoDB, populates sender usernames, and returns the history
sorted by creation time. After that, new messages arrive via Socket.IO and
are saved to the database directly inside the socket handler to avoid
redundant HTTP overhead.

The frontend architecture is a clean prop-drilling tree: Chat.jsx owns all
state and logic — users list, selected user, message history keyed by room,
socket instance, and AI state. It passes everything down to ChatLayout, which
manages the responsive 3/2/1 panel layout, then further to Sidebar,
ChatWindow, and UserInfo.

One bug I had to solve was an infinite render loop caused by the messages
array reference being unstable — the || [] fallback was creating a new array
on every render, triggering a useEffect, which set state, which caused another
render. I fixed this with useMemo to stabilize the reference.

The AI features use Google Gemini. Smart replies are on-demand — the user
clicks a sparkle button, the last received message is sent to Gemini with a
prompt asking for 3 short replies, and the results appear as clickable chips
that fill the input field. The summarize feature sends the entire formatted
conversation to Gemini and displays a 2-3 sentence summary in a banner.

If I were to take this to production, I'd add Redis for Socket.IO scaling
across multiple nodes, message pagination, JWT in httpOnly cookies instead
of localStorage, rate limiting on auth endpoints, and a proper CI/CD pipeline.
I'd also Dockerize the entire stack and add integration tests."
